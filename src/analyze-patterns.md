## 目标

分析 `data/collected/` 目录下的 Skill，提炼设计模式和最佳实践。

## 输入

- 前置条件：已执行 `collect-skills`，有 Skill 待分析
- 可选：指定分析范围（特定 Skill 或特定类别）

## 分析方法

采用**结构化范式识别方法**，按维度分解、归类、抽象三步完成分析：

1. **分解**：将 Skill 拆解为可独立分析的维度
2. **归类**：在各维度下识别相似模式，归类为范式
3. **抽象**：提炼范式特征，形成可复用的设计知识

---

## 执行步骤

### 步骤 1：读取与预处理

```
遍历 data/collected/ 目录
├── 读取每个 Skill 文件
├── 提取 frontmatter（YAML 格式）
├── 解析章节结构（按 ## 标题划分）
└── 构建结构化数据模型
```

**数据模型**：
```json
{
  "filename": "skill-name.md",
  "metadata": { "name": "", "description": "", "trigger": "" },
  "sections": ["目标", "输入", "执行步骤", "输出"],
  "content": { ... }
}
```

---

### 步骤 2：五维度分析

#### 维度 1：元数据结构（Metadata）

**分析项**：
- frontmatter 字段类型与用途
- 必选/可选字段划分
- 字段值的数据类型与约束

**检查要点**：
- [ ] 是否包含 `name`、`description`、`trigger`
- [ ] 是否有扩展字段（如 `version`、`author`、`tags`）
- [ ] `trigger` 格式是否规范（命令式 `/xxx` 或条件式）

**范式识别**：
- 基础模式：仅包含 name + description + trigger
- 扩展模式：增加 version、author、tags、dependencies 等
- 组合模式：支持多个 trigger 或条件表达式

---

#### 维度 2：文档结构（Structure）

**分析项**：
- 章节划分方式（标准章节 vs 自定义章节）
- 章节命名规范（动宾结构、名词结构）
- 章节顺序与层级关系

**标准章节对照**：
| 章节名称 | 功能定位 | 是否必需 |
|---------|---------|---------|
| 目标 | 定义 Skill 使命 | 推荐 |
| 输入 | 前置条件与数据要求 | 视情况 |
| 执行步骤 | 核心流程描述 | 必需 |
| 输出 | 结果交付物 | 推荐 |
| 注意事项 | 边界与风险提示 | 可选 |

**范式识别**：
- 标准模式：目标 → 输入 → 步骤 → 输出 → 注意事项
- 精简模式：目标 + 步骤 + 输出
- 扩展模式：增加"前提条件"、"后续操作"、"示例"等章节

---

#### 维度 3：内容设计（Content）

**分析项**：

**3.1 目标描述**
- 目标的抽象层次（高层使命 vs 具体任务）
- 是否包含量化指标
- 是否说明价值主张

**3.2 执行步骤**
- 步骤粒度（粗粒度概述 vs 细粒度操作）
- 步骤数量（3-5 步 vs 10+ 步）
- 是否包含决策分支
- 是否包含异常处理
- 是否使用伪代码或代码块

**3.3 输入输出**
- 输入定义方式（前置条件、参数、数据源）
- 输出定义方式（文件路径、返回值、副作用）
- 是否包含数据格式说明

**范式识别**：
- 声明式：描述"做什么"，不涉及具体实现
- 过程式：描述"怎么做"，包含具体操作序列
- 混合式：高层声明 + 关键步骤展开

---

#### 维度 4：触发机制（Trigger）

**分析项**：
- 触发类型：命令式（`/xxx`）、条件式（事件驱动）、混合式
- 触发粒度：文件级、项目级、全局级
- 触发时机：主动调用、自动感知、定时执行

**范式识别**：
| 触发类型 | 特征 | 适用场景 |
|---------|------|---------|
| 命令式 | 用户显式调用 `/xxx` | 工具类 Skill |
| 条件式 | 满足条件自动触发 | 自动化 Skill |
| 混合式 | 支持命令 + 条件双重触发 | 复杂 Skill |

**检查要点**：
- [ ] trigger 格式是否遵循 `/[a-z-]+` 规范
- [ ] 是否有多个 trigger 值
- [ ] trigger 命名是否语义清晰

---

#### 维度 5：质量特征（Quality）

**分析项**：

**5.1 可维护性**
- 结构是否清晰（章节划分合理）
- 描述是否一致（术语统一）
- 是否易于修改（模块化程度）

**5.2 可演进性**
- 是否支持版本标记
- 是否预留扩展点（参数化设计）
- 是否记录变更历史

**5.3 可复用性**
- 是否可独立使用
- 是否依赖特定环境
- 是否可组合使用

**质量评分标准**：
```
可维护性：⭐~⭐⭐⭐⭐⭐
可演进性：⭐~⭐⭐⭐⭐⭐
可复用性：⭐~⭐~⭐⭐⭐⭐⭐
```

### 步骤 3：范式分类与命名

将识别出的模式归类为三大范式类型：

#### 类型 1：结构范式（Structural Patterns）

定义 Skill 文件的组织方式。

| 范式名称 | 描述 | 示例 | 优势 | 劣势 |
|---------|------|------|------|------|
| Frontmatter-Body | YAML 头 + Markdown 体 | 多数 Skill | 结构清晰，易解析 | 需要 YAML 解析器 |
| Pure Markdown | 无 frontmatter，纯章节 | 简单文档 | 简单直接 | 缺少元数据 |
| Modular | 多文件组合 | 大型 Skill | 模块化，易维护 | 复杂度高 |

#### 类型 2：内容范式（Content Patterns）

定义 Skill 内容的设计方式。

| 范式名称 | 描述 | 示例 | 优势 | 劣势 |
|---------|------|------|------|------|
| Goal-Steps-Output | 目标 → 步骤 → 输出 | 标准工作流 Skill | 逻辑清晰 | 可能过于结构化 |
| Problem-Solution | 问题 → 解决方案 | 修复类 Skill | 导向明确 | 缺少过程细节 |
| Checklist | 清单式步骤 | 审查类 Skill | 可操作性强 | 缺少上下文 |

#### 类型 3：交互范式（Interaction Patterns）

定义 Skill 与用户的交互方式。

| 范式名称 | 描述 | 示例 | 优势 | 劣势 |
|---------|------|------|------|------|
| Command-Trigger | 用户输入命令触发 | `/fix-bug` | 显式明确 | 需用户记忆 |
| Context-Aware | 根据上下文自动触发 | 检测到 bug 自动建议 | 智能便捷 | 可能误触发 |
| Conversational | 通过对话引导执行 | 多轮问答式 | 灵活交互 | 效率较低 |

**分析框架**：

├── 适用场景：xxx
├── 优势：
│   ├── 点1：xxx
│   └── 点2：xxx
├── 劣势：
│   ├── 点1：xxx

**对比维度**：
- 学习成本：低/中/高
- 使用效率：低/中/高
- 灵活性：低/中/高
- 可维护性：低/中/高

## 注意事项

### 分析原则

1. **客观性原则**
   - 基于实际 Skill 案例分析，避免主观臆断
   - 范式必须有 2 个以上实例支撑
   - 劣势分析需有具体证据

2. **深度优先原则**
   - 不仅描述"是什么"，更要分析"为什么"
   - 探究设计背后的意图和权衡
   - 关注设计决策的上下文

3. **可操作性原则**
   - 范式命名简洁且易于理解
   - 提供具体可执行的建议
   - 输出报告结构化、易查阅

---

### 常见偏差及规避

| 偏差类型 | 表现 | 规避方法 |
|---------|------|---------|
| 幸存者偏差 | 只分析成功的 Skill | 收集失败案例作为反面教材 |
| 过度归纳 | 样本不足就下结论 | 至少 3 个实例才归纳范式 |
| 确认偏差 | 只找支持预设的证据 | 主动寻找反例和边界情况 |
| 近期偏差 | 过度关注最近分析的 Skill | 随机打乱分析顺序 |

### 分析工具建议

可使用以下工具辅助分析：

```bash
# 统计 Skill 数量
find data/collected -name "*.md" | wc -l

# 提取所有 frontmatter
for f in data/collected/*.md; do
  sed -n '/^---$/,/^---$/p' "$f"
done

# 查找特定章节
grep -r "^## 目标" data/collected/

# 统计章节分布
grep -r "^## " data/collected/ | cut -d: -f2 | sort | uniq -c
```
