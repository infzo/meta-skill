# Skill 设计原则

> 本文档沉淀优秀 Skill 的设计模式，持续演进更新。

## 原则分类

### 一、结构设计原则

#### 1.1 Frontmatter-Body 分离原则

**发现来源**：所有收集的 Skill

**原则内容**：
Skill 文件应采用 YAML frontmatter + Markdown body 的标准结构。frontmatter 存放机器可读的元数据，body 存放人类可读的内容。

**应用场景**：所有 Skill 文件

**示例**：
```markdown
---
name: skill-name
description: 一句话描述
trigger: /skill-name
---

## 目标
...
```

---

#### 1.2 标准章节划分原则

**发现来源**：多数收集的 Skill

**原则内容**：
Skill 应包含以下标准章节：
- **目标**：明确 Skill 要达成什么
- **输入**：前置条件和数据要求
- **执行步骤**：核心流程描述
- **输出**：交付物和存储位置
- **注意事项**：边界和风险提示（可选）

**应用场景**：工作流类 Skill

**示例**：
- 正例：目标 → 输入 → 执行步骤 → 输出 → 注意事项
- 反例：仅有执行步骤，缺少目标和输出定义

---

### 二、内容设计原则

#### 2.1 目标明确化原则

**发现来源**：Self-Improving Agent, Proactive Agent

**原则内容**：
目标描述应回答三个问题：
1. 这个 Skill 做什么？
2. 为什么需要它？
3. 什么时候使用？

**应用场景**：所有 Skill

**示例**：
```markdown
## 目标
捕获经验教训、错误和纠正，以实现持续改进。

使用时机：
（1）命令或操作意外失败
（2）用户纠正 Agent 的输出
...
```

---

#### 2.2 步骤粒度适中原则

**发现来源**：Agent Browser, Find Skills

**原则内容**：
执行步骤应：
- 粗粒度概述 + 关键细节展开
- 每个步骤可独立执行
- 包含决策分支（如有必要）
- 明确错误处理方式

**应用场景**：复杂工作流 Skill

---

#### 2.3 输入输出明确化原则

**发现来源**：所有收集的 Skill

**原则内容**：
- 输入：区分必需输入和可选输入，提供默认值
- 输出：明确文件路径、格式、存储位置

**应用场景**：所有 Skill

---

### 三、触发设计原则

#### 3.1 命令命名规范原则

**发现来源**：所有收集的 Skill

**原则内容**：
- 使用 kebab-case 命名
- 语义清晰，见名知意
- 避免与系统命令冲突

**应用场景**：所有 Skill

**示例**：
- 正例：`/self-improving-agent`, `/find-skills`
- 反例：`/SelfImproving`, `/findSkills`

---

#### 3.2 触发时机说明原则

**发现来源**：Find Skills, Skill Vetter

**原则内容**：
Skill 应说明触发时机：
- 主动触发（用户调用）
- 被动触发（条件满足）
- 定时触发（周期执行）

**应用场景**：自动化类 Skill

---

### 四、质量设计原则

#### 4.1 可维护性原则

**发现来源**：所有收集的 Skill

**原则内容**：
- 结构清晰，章节划分合理
- 术语统一，避免歧义
- 预留扩展点

**应用场景**：长期维护的 Skill

---

#### 4.2 可演进性原则

**发现来源**：Self-Improving 系列

**原则内容**：
- 支持版本标记
- 记录变更历史
- 参数化设计，便于调整

**应用场景**：需要持续优化的 Skill

---

#### 4.3 可组合性原则

**发现来源**：Find Skills, Ontology

**原则内容**：
- Skill 可独立使用
- 可与其他 Skill 组合
- 避免强依赖特定环境

**应用场景**：工具类 Skill
