# Skill 质量检查清单

> 用于验证 Skill 文件的质量和完整性。

---

## 1. 完整性检查

### 1.1 元数据完整

- [ ] frontmatter 包含 `name` 字段
- [ ] frontmatter 包含 `description` 字段（一句话描述）
- [ ] frontmatter 包含 `trigger` 字段（命令格式：`/xxx`）
- [ ] 各字段值不为空

### 1.2 目标明确

- [ ] 有明确的"目标"章节
- [ ] 目标可测量、可验证
- [ ] 包含使用时机说明（可选）

### 1.3 输入输出清晰

- [ ] 输入项已列出
- [ ] 区分必需输入和可选输入
- [ ] 可选输入有默认值说明
- [ ] 输出项已定义
- [ ] 输出位置明确（文件路径或返回值）

### 1.4 步骤覆盖完整

- [ ] 执行步骤已定义
- [ ] 步骤顺序合理
- [ ] 步骤粒度适中（不是过于粗略或过于细碎）

---

## 2. 可执行性检查

### 2.1 步骤可独立执行

- [ ] 每个步骤描述具体操作
- [ ] 步骤不依赖未定义的前置条件
- [ ] 步骤间依赖关系明确

### 2.2 无循环依赖

- [ ] 无步骤 A → B → A 的循环
- [ ] 无跨阶段循环依赖

### 2.3 错误处理覆盖关键失败点

- [ ] 识别了关键失败点
- [ ] 每个失败点有处理方案
- [ ] 错误处理方案可执行

---

## 3. 可维护性检查

### 3.1 命名语义化

- [ ] Skill 名称语义清晰（见名知意）
- [ ] 使用 kebab-case 命名
- [ ] 步骤名称语义化

### 3.2 结构清晰

- [ ] 章节划分合理
- [ ] 层级不超过 4 级
- [ ] 使用标准章节命名

### 3.3 有版本控制和更新记录（长期维护 Skill）

- [ ] 包含更新日志章节
- [ ] 或有版本号标记

### 3.4 包含使用示例（推荐）

- [ ] 提供具体使用示例
- [ ] 示例可直接复制使用

---

## 4. 最佳实践检查

### 4.1 触发条件合理

- [ ] trigger 命名不与系统命令冲突
- [ ] trigger 格式规范：`/[a-z-]+`
- [ ] 如有条件触发，触发条件明确

### 4.2 单一职责

- [ ] Skill 聚焦单一目标
- [ ] 功能范围适度，不过于庞大

### 4.3 可组合

- [ ] Skill 可独立使用
- [ ] 可与其他 Skill 组合（如适用）
- [ ] 输出可作为其他 Skill 输入

### 4.4 有回滚机制（涉及修改操作）

- [ ] 涉及修改的 Skill 有回滚方案
- [ ] 或有备份机制

---

## 5. 格式检查

### 5.1 Markdown 格式正确

- [ ] 标题层级正确
- [ ] 列表格式正确
- [ ] 代码块有语言标记

### 5.2 无格式错误

- [ ] 无未闭合的代码块
- [ ] 无格式混乱的表格

---

## 检查结果判定

| 检查项数 | 通过标准 | 结果 |
|---------|---------|------|
| 完整性（4项） | ≥ 4 | ✅/❌ |
| 可执行性（3项） | ≥ 3 | ✅/❌ |
| 可维护性（4项） | ≥ 3 | ✅/❌ |
| 最佳实践（4项） | ≥ 3 | ✅/❌ |
| 格式检查（2项） | ≥ 2 | ✅/❌ |

**总体判定**：所有类别 ≥ 通过标准 → ✅ 通过
